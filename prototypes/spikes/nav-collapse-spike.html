Nav-collapse spike — CONSTITUTION.md §6

What this proves: a responsive nav that collapses into a disclosure pattern below 860px, implemented entirely with the native HTML popover attribute + popovertarget invoker — the same mechanism terminus itself uses at the pinned snapshot (templates/partials/menu.html). Zero JavaScript, unlike terminus's own close_responsive_menu_on_resize auto-closer, which this deliberately does not carry forward (per CONSTITUTION.md §2/§6).

How to test: resize this window (or your devtools responsive-mode viewport) across 860px. Above it, all 8 items show inline. Below it, they collapse behind a "Menu" button. Try keyboard-only: Tab to the button, Enter/Space to open, Tab through the items, Escape to close (native "auto" popover light-dismiss — no JS). Click outside the open panel to confirm it also closes without JS.

The nav below intentionally has 8 items of realistic length — none of the 25 style-tile prototypes tested this, since they all used the same 4 short items ("Research", "Blog", "CV"...), which happened to never overflow. This spike exists specifically to catch that gap.

Checklist

  • ✅ No horizontal overflow at any width, including with 8 items (vs. the 25 prototypes' untested 4-item navs).
  • ✅ Zero JavaScript: open/close is native popovertarget behaviour; light-dismiss (outside-click, Escape) is native to popover="auto" (the implicit default).
  • ✅ Keyboard operable: the toggle is a real <button> (native Tab/Enter/Space semantics), and items inside an open popover are reachable via Tab since the browser places the popover in the top layer.
  • ✅ Above the breakpoint, the exact same markup renders as a normal expanded nav — no separate "desktop-only" markup branch needed.

Breakpoint decided by this spike: 860px (chosen because 8 realistic-length nav items need roughly that much horizontal space before wrapping/crowding at typical body/nav font sizes — re-check per variant if a style's nav font is significantly larger/smaller).
